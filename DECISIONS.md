## D-001. 실시간 채팅: Supabase Realtime 사용 (직접 WebSocket 서버 미구현)

**결정일:** 2026-06  
**상태:** 확정 (v1 기준)

### 결정 내용

실시간 채팅 구현에 **Supabase Realtime**을 사용한다. 직접 WebSocket 서버를 구현하지 않는다.

### 배경 — Supabase Realtime과 WebSocket의 관계

Supabase Realtime은 **내부적으로 WebSocket 기반 실시간 통신**을 사용한다. 클라이언트는 `supabase.channel(...).on('postgres_changes', ...)` API로 DB 변경 이벤트를 구독하며, 이 연결은 Supabase가 관리하는 WebSocket 서버를 통해 이루어진다. 따라서:

- **WebSocket 통신은 실제로 일어난다** — 브라우저 DevTools에서 `wss://` 연결을 확인할 수 있다.
- **그러나 직접 WebSocket 서버를 작성·운영하지 않는다** — Supabase가 서버 역할을 대신한다.
- 이 둘은 다르다. 직접 구현이란 Node.js(`ws`, `socket.io` 등)로 서버를 작성하고, 연결 관리·메시지 라우팅·방(room) 관리를 직접 코드로 다루는 것을 의미한다.

### 이번 선택의 이유

| 요소 | 판단 |
|------|------|
| 개발 기간 | 1개월. 직접 WebSocket 서버 구현 시 서버 운영·인증 연동·스케일링 고려까지 필요 → 일정 초과 위험 |
| 운영비 | 직접 서버 = 별도 호스팅 비용 발생. Supabase Realtime은 무료 티어 내 |
| 학습 목표 충족 여부 | Supabase Realtime을 통해 **실시간 채팅의 핵심 개념**(구독 패턴, 이벤트 처리, 낙관적 업데이트, 클라이언트-서버 동기화)을 충분히 학습할 수 있다 |
| 실사용 완성 | v1을 실제 여행에 투입하는 것이 목표 → 기능 안정성 우선 |

### 학습 측면

채팅 기능은 이번 프로젝트의 핵심 학습 목표 중 하나다. Supabase Realtime을 사용하더라도 다음을 직접 다루게 된다:

- WebSocket 채널 구독/해제 생명주기 관리 (`useEffect` + cleanup)
- 실시간 이벤트를 React Query 캐시에 머지하는 패턴
- 낙관적 업데이트 + 실패 시 롤백
- 2개 클라이언트 간 메시지 동기화 확인

이 경험을 바탕으로 **직접 WebSocket 서버 구현은 후속 학습 과제로 분리**한다.

### 후속 학습 과제 (v1 이후)

- Node.js + `ws` 또는 `socket.io`로 WebSocket 서버 직접 구현
- 방(room) 관리, 브로드캐스트, 재연결 처리 직접 작성
- Supabase 없이 순수 WebSocket만으로 동일한 채팅 기능 재현

## D-002. 채팅 MVP 범위 명확화

**결정일:** 2026-06  
**상태:** 확정 (v1 기준)

### MVP 포함 (Week 3 필수 구현)

- 텍스트 메시지 전송
- Supabase Realtime으로 실시간 수신 (새로고침 없이 도착)
- 내 메시지 / 상대 메시지 구분 (버블 방향·색상)
- 시간 표시

### Backlog (v1 이후)

- 이미지 첨부 (Supabase Storage 연동)
- 읽음 표시 (read receipt)
- 입력 중 표시 (typing indicator)
- 고급 무한스크롤 (과거 메시지 페이지네이션) — v1은 최근 N건 초기 로드로 대체
- 웹 푸시 알림 (iOS PWA 제약 있음)

### 컷 정책

채팅은 **학습 목표이자 실사용 핵심 기능**이므로 일정이 밀려도 컷하지 않는다.  
일정 압박 시 메모 → PWA 다듬기 순으로 먼저 컷한다.
