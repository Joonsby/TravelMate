# 📋 PROJECT_PLAN.md — 우리 여행 앱 개발 계획

> **목표:** 1인 개발 / 1개월 / **실제 여행 출발 전까지 쓸 수 있는 v1** 완성
> **스택:** React + Vite + TypeScript · Supabase · Vercel · PWA
> 설계 근거: [`ARCHITECTURE.md`](./ARCHITECTURE.md)

---

## 0. 전제 & 성공 기준

이 계획의 단 하나의 성공 기준:

> **여행 출발일까지, 둘이 실제 폰에 설치해서 일정·체크리스트·경비를 쓸 수 있다.**

그래서 "기능을 더 넣는 것"보다 **"여행에 꼭 필요한 것을 빠짐없이 끝내는 것"** 을 우선한다.
화려한 부가기능(드래그 정렬, 다크모드, 푸시 등)은 전부 출시 후(Backlog)로 미룬다.

## 1. MVP 범위

### ✅ 포함 (In Scope)
| 영역 | 최소 구현 |
|------|-----------|
| 인증 | 2계정 로그인, 커플 단위 데이터 공유(RLS) |
| 여행 | 여행 1개 생성·선택 |
| 일정 | 일자별 아이템 추가/수정/삭제 (시간순 표시) |
| 체크리스트 | 항목 추가/체크/삭제, 진행률 |
| 경비 | 지출 기록 + 카테고리 합계 + **2인 정산 요약** |
| 채팅 | 텍스트 전송 · 실시간 수신 · 내/상대 메시지 구분 · 시간 표시 |
| 메모 | 작성/수정/삭제 |
| 인프라 | Vercel 배포 · PWA 설치 · keep-alive cron |

### ❌ 제외 (Out of Scope, 이번엔 안 함)
- 이미지 첨부(채팅 사진·영수증 사진) → Backlog
- 채팅 읽음 표시, 입력 중 표시(typing indicator) → Backlog
- 채팅 고급 무한스크롤(페이지네이션, 최근 N건 초기 로드로 대체) → Backlog
- 직접 WebSocket 서버 구현 → 후속 학습 과제(DECISIONS.md 참고)
- 여러 여행 동시 관리 UI(스위처 고도화) → 단순 목록까지만
- 일정 드래그 정렬 → 시간순 자동 정렬로 대체
- 푸시 알림, 다크모드, 환율 계산, 지도 연동
- 회원가입 UI (초기 2계정은 Supabase 대시보드에서 수동 생성)

> 💡 **현실 안전장치:** 만약 일정이 밀리면 **메모·PWA 다듬기를 먼저 컷**한다. **채팅은 이번 프로젝트의 핵심 학습 목표(Realtime/WebSocket 개념)이므로 컷하지 않는다.** 여행에 필요한 **일정·체크리스트·경비·채팅**은 절대 컷하지 않는다.

## 2. 기능 우선순위

여행 직접 필요도 × 구현 난이도로 정렬. P0가 가장 먼저·반드시.

| 우선순위 | 기능 | 이유 |
|----------|------|------|
| **P0** | 셋업 · 인증 · DB/RLS · 배포 골격 | 이게 없으면 아무것도 못 함 |
| **P0** | 일정 | 여행의 핵심. 출발 전 반드시 |
| **P0** | 체크리스트 | 준비물 — 출발 전 반드시 |
| **P0** | 경비 + 정산 | 여행 중·후 반드시 |
| **P0.5** | 채팅(실시간) | 핵심 학습 목표(Supabase Realtime / WebSocket 개념) + 실사용 필수. 컷 불가 |
| **P1** | 메모 | 있으면 좋음. 시간 부족 시 컷 가능 |
| **P2** | PWA 설치 다듬기, 빈 상태/에러 처리 | 완성도. 시간 부족 시 컷 가능 |

## 3. 주차별 개발 일정

원칙: **매주 끝에 "배포된 무언가"가 있다.**

### Week 1 — 기반 + 인증 (P0)
셋업·DB·인증·배포 골격을 끝낸다. 빈 화면이라도 배포까지.
- 산출물: 로그인 → 빈 탭 화면, 프로덕션 URL 존재, keep-alive 작동

### Week 2 — 일정 + 체크리스트 (P0)
여행에 꼭 필요한 두 기능 완성.
- 산출물: 여행 만들고 일정·체크리스트를 실제로 채울 수 있음

### Week 3 — 채팅(반드시) + 메모 (P0.5/P1)
채팅은 이번 프로젝트의 핵심 학습 목표(Supabase Realtime · WebSocket 개념)이므로 반드시 이 주차에 완성한다. 메모는 시간 남으면 진행.
- 산출물: 둘이 실시간 대화 가능 (채팅 필수 · 메모는 시간 남으면)

### Week 4 — 경비(초반 즉시) + 마무리 + 출시 (P0/P2)
Week 4 시작 즉시 경비 착수. 경비가 밀리지 않도록 Week 4 초반에 완성 후 테스트·정식 배포로 이어진다.
- 산출물: **실여행 투입 가능한 v1 출시**

> ⚠️ Week 3 메모가 밀렸을 경우 → Week 4 초반 경비 완성 후 잔여 시간에 메모를 처리한다. 채팅과 경비는 어떤 경우에도 컷하지 않는다.

## 5. 완료 조건 (Definition of Done)

### 작업(Issue) 단위 DoD
하나의 Issue는 아래를 **모두** 만족해야 닫는다.
- [ ] 기능이 의도대로 동작한다 (수용 기준 충족)
- [ ] TypeScript 타입 에러 없음 / `pnpm lint` 통과
- [ ] **모바일 화면(실기기 또는 모바일 뷰)** 에서 확인했다
- [ ] 데이터 접근은 RLS를 통과한다 (남의 데이터 안 보임)
- [ ] 로딩·빈 상태·실패 시 화면이 깨지지 않는다
- [ ] `feature/*` 브랜치 → PR → `main` 머지 → Vercel 배포 반영 확인

### MVP(v1) 단위 DoD — "출시해도 된다"
- [ ] P0 기능(인증·일정·체크리스트·경비) 전부 동작
- [ ] 두 사람이 **각자 폰**에서 로그인해 같은 데이터를 본다
- [ ] 한쪽이 입력하면 다른 쪽에서 (새로고침으로라도) 보인다
- [ ] 프로덕션 URL이 살아있고 PWA로 설치된다
- [ ] keep-alive cron이 작동해 7일 후에도 앱이 멈추지 않는다
- [ ] 치명 버그(로그인 불가·데이터 유실·화면 크래시) 0건

## 6. 출시 전 테스트 체크리스트

여행 출발 **최소 3일 전**에 아래를 끝낸다. (문제 발견 시 고칠 시간 확보)

### 인증 / 권한
- [ ] 두 계정 모두 로그인/로그아웃 정상
- [ ] 새로고침 후 세션 유지
- [ ] 다른 사용자의 데이터가 보이지 않음(RLS) — 가능하면 더미 계정으로 교차 확인

### 기능별 (각각 추가→수정→삭제 1회)
- [ ] 여행 생성·선택
- [ ] 일정 아이템 추가/수정/삭제, 시간순 정렬 확인
- [ ] 체크리스트 추가/체크/삭제, 진행률 갱신
- [ ] 경비 추가/삭제, 카테고리 합계, **정산 금액 수기 검산 일치**
- [ ] 채팅 전송/수신(두 기기 동시 켜고 실시간 도착)
- [ ] 메모 작성/수정/삭제

### 동기화 / 실시간 / 오프라인
- [ ] A기기 입력 → B기기에서 확인됨
- [ ] 채팅이 새로고침 없이 도착(Realtime)
- [ ] 약한 네트워크(또는 비행기 모드 잠깐)에서 입력 시 앱이 죽지 않음, 복구 후 반영

### PWA / 인프라
- [ ] iOS Safari, Android Chrome 각각 홈 화면 설치
- [ ] 설치 후 standalone(주소창 없이) 실행
- [ ] keep-alive 수동 1회 실행 성공, 로그 확인
- [ ] **데이터 백업 1회**(Supabase에서 수동 export) — 무료 티어는 자동 백업 없음

### 최종
- [ ] 여행 기간을 커버하는 더미 여행으로 리허설 1회(일정·체크·경비 한 바퀴)
- [ ] 두 사람 모두 로그인된 상태로 폰에 설치 완료
